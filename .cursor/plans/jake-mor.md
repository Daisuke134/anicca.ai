# Jake Mor's 100 Tips for iOS Subscription App Growth

## 📊 カテゴリー1: 最重要メトリクス（#1-5）

### #1 — App Install → Paywall View が最重要メトリクス
**核心**: アプリインストールからPaywall表示までのコンバージョン率を追跡すべき。このメトリクスと売上は**ほぼ線形関係**にある。

> 💡 **Aniccaへの適用**: `AppState`でPaywall表示回数を追跡し、Mixpanelに送信する仕組みを追加すべき。

### #2 — 未購読ユーザーあたりのPaywall表示回数
**核心**: 平均して**7回目のPaywall表示まで収益低下が見られない**。つまり、何度もPaywallを見せても大丈夫。

> 💡 **Aniccaへの適用**: 現在1回のPaywall表示だけでなく、複数回表示する戦略を検討。

### #3 — 最適な初期セットアップ
```
✅ オンボーディング前にPaywall
✅ オンボーディング後にPaywall  
✅ アプリ起動のたびにPaywall
```

### #4 — 毎日プッシュ/ローカル通知を送信
**ロジック**: 各アプリ起動 = Paywall表示（#3に従えば）→ Paywall表示増加 = 売上増加（#2より）

### #5 — 24時間リテンションと7日リテンション
収益化に注力する際、これらのメトリクスを意思決定に活用。どの程度のリテンション低下を許容して売上増加を取るか判断する。**価格が高すぎて離脱するユーザーは、どのみちリテンションしない傾向がある**。

## 💳 カテゴリー2: Web Checkout戦略（#6-9）

### #6 — Web Checkoutは最強のツール
**用途**: 有料UA（ユーザー獲得）ではなく、**購読しなかったインストールへのマーケティング割引**に使う。

```
appname.com/checkout?app_user_id=100
↓
購入後、アプリにディープリンク
```

> 💡 **Aniccaへの適用**: Stripeを使ったWebチェックアウトページを作成し、未購読ユーザーへの割引キャンペーンに活用できる。

### #7 — ドリップキャンペーンを実行
| 日数 | アクション |
|------|-----------|
| Day 0 | 最初の割引 |
| Day 3 | リマインダー |
| Day 7 | さらに割引 |
| Day 14 | 追加オファー |
| Day 30 | 最終オファー |

プッシュ、SMS、メールでリマインドし、Web checkoutにリンク。**30%オフから始めて、時間とともに割引を深くする**。

### #8 — 最初の割引通知のタイミング決定
**分析方法**: トライアルの95%がどのくらいで来るか分析（通常1時間）。その後にStripe checkoutリンクを30%オフで送信。

### #9 — Stripeのリテンションはアプリ内課金の約2倍
**驚くべき事実**: Stripeでの購読は継続率が2倍高い。
- ダークパターンは使わない
- 設定でキャンセル方法を提供
- 領収書をメールで送信

## 💰 カテゴリー4: 価格テスト（#11-18）

### #11 — 価格テストの心得
- 売上増加は通常10%以下なので、**やりすぎない**
- まず週額→月額→年額の順でテスト
- コホートを時間をかけて分析

### #12 — 週額価格のテスト候補
```
$1.99, $2.99, $3.99, $4.99, $7.99, $9.99
```
3日または7日のトライアルをテスト

### #13 — 月額価格のテスト候補
```
$4.99, $7.99, $9.99, $12.49, $14.99, $19.99, $29.99, $49.99
```

### #14 — 年額価格のテスト候補
```
$29.99, $39.99, $49.99, $59.99, $89.99, $119.99, $129.99, $149.99, $189.99
```

### #15 — 価格が特定の値である理由
これらはランダムではない。Superwallで勝利する価格であり、**週/月換算がきれいになる**：
- $119.99/年 = 月$10
- $129.99/年 = 週$2.49

> 💡 **ユーザーにこれを伝える**: 「月額たったの$10」

### #16 — 年額を決めたら固定
月額・週額のコホートを時間をかけて分析。**年間購読開始から13ヶ月後に決断**。13ヶ月の週/月額 vs 2回の年額更新を比較。

### #17 — 複数プロダクトを提供する場合
```
トライアルなし月額: $10/月
年額（7日トライアル付き）: $59.99/年（= $5/月、50%オフとして表示）
```
年額の方が2倍お得に見せる。

### #18 — 組み込みWin-backキャンペーン
**秘密の割引プラン**をサブスクリプショングループに隠す。

```
通常: $89.99/年（アプリ内表示）
秘密: $59.99/年（"Secret Discount"という商品名）
```

ユーザーがキャンセルしようとすると、App Storeが自動的に割引を表示。**売上10%増加の事例あり**。

> 💡 **Aniccaへの適用**: App Store Connectで低価格の秘密プランを追加し、キャンセル時のWin-back Offerとして設定。

## 📋 カテゴリー5: サーベイ戦略（#19-25）

### #19 — サーベイは重要
**必須条件**:
1. 結果を時間経過で追跡できること
2. 質問を簡単に変更できること

### #20 — 最もシンプルなセットアップ
アプリ起動時にTypeformサーベイを表示。`app_user_id`を隠しパラメータとして追加。

### #21 — Webhookでデータ同期
TypeformからMixpanel/Amplitudeにwebhookで結果を送信。既存ツールで追跡可能に。

### #22 — 2種類のサーベイ
1. **トライアルキャンセルサーベイ**
2. **PMF（Product Market Fit）サーベイ**

RevenueCat + プッシュ通知サービスでトライアルキャンセルサーベイをトリガー。

### #23 — トライアルキャンセルサーベイの質問
```
1. なぜキャンセルしましたか？（選択式）
2. どう改善できますか？（自由回答）
→ 最後にStripe checkoutで割引を提示
```
自由回答の分析には**ワードクラウド**を使用。

### #24 — PMFサーベイ
コア機能を3回使用した後にアプリ内で質問。[@rahulvohra](https://twitter.com/rahulvohra)のSuperhuman PMFフレームワークを使用。

参考: [Superhuman's Product Market Fit Framework](https://review.firstround.com/how-superhuman-built-an-engine-to-find-product-market-fit)

### #25 — レビューを大量に獲得する方法
PMFサーベイで「なぜこのアプリが好きですか？」と質問。
最後に**回答を見せて**、「これをレビューとして貼り付けてください」と依頼。

Typeformのリコール機能でExit pageでApp Storeにリンク。

## 🎓 カテゴリー7: オンボーディング最適化（#28-30）

### #28 — オンボーディング最適化のバランス
```
完了率 ↔ Paywallコンバージョン率
```
- 高い完了率 = Paywall表示回数増加
- 長いオンボーディング = Paywallコンバージョン率向上（**サンクコスト効果**）

### #29 — 完了率が最も低いステップ
```
❌ 名前、年齢、メール、電話番号
❌ アプリ権限
```
→ **Paywall表示の後に移動**

> 💡 **Aniccaへの適用**: 現在のオンボーディングフローでプロフィール入力はPaywall前にあるが、順序を検討する価値あり。

### #30 — オンボーディングステップの並び替え
各ステップを**完了率の降順**で並べる。

例：ステップ4→5が99.5%で最高なら、それを最初に。最低完了率のステップは最後（サンクコスト効果で完了率が上がる）。

## 🎨 カテゴリー8: Paywall最適化（#31-39）

### #31 — 最初のPaywallテスト = メッセージング
```
✅ ヘッドライン
✅ 短い段落
✅ 機能チェックリスト
✅ 1つの購入ボタン（下部）
❌ 画像なし
❌ スクロールなし
❌ その他一切なし
```
これが機能開発からPaywall改善まで全てに影響。

### #32 — Paywallコピーのテストは機能開発より簡単
**まだ作っていない機能をPaywallで匂わせてもOK**（Teslaの「Full Self Driving」のように）。

リフトが見られたら、その機能を優先開発。

### #33 — メッセージングが決まったらデザイン
Paywallが購入なしで閉じられたら、**なぜ？の自由回答サーベイ**を表示。
回答から選択式質問を作成。ワードクラウドで分析。

### #34 — #33の上位5回答に対応
恐れを軽減するグラフィック/画像を作成。

例: Blinkistのフリートライアルタイムライン（でも全アプリに効くわけではない）

### #35 — 具体例
| ユーザーの不安 | 対策 |
|---------------|------|
| 無料と有料の違いがわからない | 比較テーブルを追加 |
| いつ課金されるかわからない | トライアルタイムラインを追加 |

### #36 — Superwallで常に効果があるもの：ビデオPaywall
オンボーディング前に**画面録画ビデオ**を表示。
- ユーザーにトライアルで何が得られるか正確に見せる
- 3D Apple風グラフィックを追加
- **Rotato**で作成すると良い

### #37 — Paywallアイデアの情報源
[paywallscreens.com](https://paywallscreens.com/) — Superwallが買収し、7,200以上のPaywallを収集。Sensor Tower売上予測も含む。

### #38 — リモートでPaywallを更新できることが必須
Superwallが最も簡単だが、唯一の方法ではない。

### #39 — Firebase Remote Configを使う方法
価格、テキスト、画像をリモートで変更。
**画像をUIエレメントとして使用**すれば、完全にリモート更新可能なPaywallを構築できる。

## 🔍 カテゴリー11: ASO（App Store Optimization）（#50-59）

### #50 — ASOにはApple Search Ads（ASA）を使う
目的: **高意図ユーザーを見つけるキーワード発見**（必ずしも収益性ではない）。

AppFigures/AppTweakで100-200の関連キーワードリストを作成。

### #51 — キーワードをキャンペーンに追加
**収益性は気にしない**。それは目的ではない。
キーワードアトリビューションをMixpanelに送信し、インストール→課金コンバージョンと30日リテンションを確認。

### #52 — タイトルと副題の前方にキーワード
最初のキーワードが最も重み付けが高い。
**ブランドより実用性を重視**。

### #53 — アプリタイトルの形式
```
"キーワード - アプリ名"
例: "Personal Trainer - FitnessAI"
```

### #54 — 副題にキーワード
文法は気にしない。
```
"weight lifting exercise workout"
```
これでOK。

### #55 — キーワードリストの決定
**短いキーワードに高スコアを与える**（他のキーワードの余地を作る）。
スペースなし、カンマのみ。順番に配置。

### #56 — スコアリング公式の例
```
スコア = (インストール→課金率 × D30リテンション × 人気度) / (キーワード長 × 競争度)
```
最大化したい値は分子に、最小化したい値は分母に。

### #57 — App Store Connectでアプリ内イベント作成
タイトルは検索結果にオーガニックランクイン可能。
**裏技**: 開始日を提出後約1時間に設定すると、アプリ審査が**早まる**。

### #58 — タイトル変更しても古いキーワードランキングを維持
四半期に1回程度変更。効果が減るまで続ける。

### #59 — App Storeストーリーの活用
アプリのストーリーが書かれたら、タイトル変更でストーリーのタイトルも変わる。
ストーリーもアプリと同様に古いキーワードでランキング維持。
